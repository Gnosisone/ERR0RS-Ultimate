What is ERR0RS?

01 // Prompt Anatomy
Every effective ERR0RS prompt has a structure. Understanding each part lets you get exactly what you need without burning tokens on vague back-and-forth.
The Five-Part Prompt Structure
ROLE
Who ERR0RS should act as. "Act as a red team operator..." Sets the mindset and output style for the whole session.
CONTEXT
What environment, scope, or target you're working with. Always include OS, network range, authorization scope.
TASK
The specific action you want. Be precise. "nmap SYN scan all 65535 ports, service version detection, XML output" is strong.
FORMAT
How you want output delivered. "Command first, then explain each flag, then expected output."
CONSTRAINTS
Limits or edge cases. "Do not touch port 443, avoid noisy scans, stay stealth."
Weak vs Strong Prompt
WEAK — avoid this
USER > "scan the network" RESULT: Ambiguous. Which network? What kind of scan? ERR0RS has to guess — guesses waste time and may violate scope.
STRONG — this is the way
USER > "Act as red team recon operator. Authorized test on 192.168.10.0/24. Stealth SYN scan top 1000 ports, skip host discovery, grepable output to ~/recon/targets.gnmap. Then show which hosts have port 80 or 443 open." RESULT: Precise. Scoped. Formatted. ERR0RS executes clean.
// TOKEN TIPFirst prompt in a session should always carry full context. Short follow-ups are fine after that. On Pi 5 7B, keep prompts under 512 tokens for snappy response.

04 // Prompt Techniques
Battle-tested techniques that get the best results out of ERR0RS. Each has a specific use case — don't use a sledgehammer when you need a scalpel.
Technique 01 — Chain-of-Thought (CoT)

Force ERR0RS to reason step-by-step before executing. Critical for complex attacks where order of operations matters.

// TRIGGERAppend: "Think step by step before giving me the commands."
CoT Example
USER > "Foothold as www-data on 10.10.10.50. Think step by step about privesc. Consider SUID, sudo misconfigs, cron, writable paths, kernel exploits. Low-noise first." ERR0RS reasons through each vector before recommending.
Technique 02 — Few-Shot Examples

Show ERR0RS an example of what you want, then ask it to repeat for your target. Most effective way to control output format.

# Few-shot template EXAMPLE INPUT: target is Apache 2.4.49 EXAMPLE OUTPUT: CVE-2021-41773, path traversal + RCE, searchsploit -x apache/webapps/50383.sh NOW DO THIS: target is OpenSSH 8.1 # ERR0RS matches your example format exactly
Technique 03 — Negative Prompting

Tell ERR0RS what NOT to do. Critical for staying within rules of engagement.

// EXAMPLES "Do NOT use Metasploit — manual only."
"Do NOT generate payloads — explain the concept."
"Do NOT touch 192.168.1.1 — that's the router."
"Do NOT run anything loud — IDS is monitoring."
Technique 04 — Output Templating

Specify an exact structure for ERR0RS's response. Useful when piping output into reports or the RAG system.

"Give me output in this exact format: TOOL: [tool name] COMMAND: [exact command] FLAGS: [each flag explained] EXPECTED OUTPUT: [what you'll see] OPSEC RATING: [Low/Medium/High noise] DETECTION: [how blue team catches this]"
Technique 05 — Iterative Refinement

Run one stage, show ERR0RS the output, ask for the next step. This mimics how real engagements work.

Iterative Session
STEP 1 > "Run recon on 10.10.10.5" ERR0RS: [nmap results returned] STEP 2 > "Port 21 open, vsftpd 2.3.4. What's the play?" ERR0RS: [vsftpd backdoor, MSF module details] STEP 3 > "Got shell as root. Post-exploitation checklist." ERR0RS: [hashdump, persistence, loot, cleanup]
Technique 06 — Constraint Injection
// TEMPLATE"Authorized internal pentest for [Company]. Scope: [IPs]. Time box: [X hours]. Client priority: stealth over speed. RoE: no destructive payloads, no DoS. Report format: CVSS scored."

06 // Common Prompting Errors
The most common mistakes operators make when talking to ERR0RS — and how to fix them fast.
Error 01 — Context Amnesia
// PROBLEMStarting a new session without giving ERR0RS context. The model has no memory between sessions — it doesn't know your target, OS, or scope unless you tell it.
// FIXKeep a session header snippet you paste at the start of every engagement. 10 seconds of setup saves 10 minutes of confusion.
Error 02 — One-Word Prompts
// PROBLEM"scan" / "exploit" / "shell" — Too vague. ERR0RS asks clarifying questions or gives a generic response.
// FIXMinimum viable prompt: [action] + [target] + [constraint]. Example: "Scan 10.0.0.5 (authorized) for open ports — stealth, no ping, top 1000."
Error 03 — Asking for Everything at Once
// PROBLEM"Run recon, find vulns, exploit, escalate, dump creds, write report for 10.0.0.0/24." Overwhelms context window, produces rushed output.
// FIXBreak it into stages. Each stage gets full attention. Use the Campaign Manager for multi-stage orchestration.
Error 04 — Wrong Model for the Job
// PROBLEMRunning 7B on Pi 5 for a complex AD kill chain analysis. 7B can't hold enough context for deep multi-step reasoning.
// FIX7B → quick commands, single-tool help, flag lookups. 32B → kill chain planning, deep reasoning, malware analysis.
Error 05 — Not Showing ERR0RS Your Output
// PROBLEM"The scan finished." ERR0RS can't help if it doesn't see what you found.
// FIXAlways paste relevant output back. "Here's the nmap result: [paste]. What are the most promising vectors?" This is how the ReAct loop is supposed to work.
Error 06 — Ignoring OPSEC Flags
// FIXWhen ERR0RS gives an OPSEC rating, trust it. If you need speed over stealth, say so explicitly: "Speed over stealth — client authorized noisy testing."
